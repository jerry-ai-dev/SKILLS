## 一、核心概念速查表

| 概念 | 一句话定义 | 生活类比 |
|------|-----------|---------|
| GRPO | Group Relative Policy Optimization：用组内相对奖励替代 Critic 的 PPO 变体 | 考试不看绝对分，只看你在这组里排第几 |
| 组内归一化优势 $\hat{A}_i$ | $(r_i - \text{mean}) / \text{std}$，表示这条回答比组内平均好多少 | 你的考试分减去班级平均分再除以标准差 = Z-score |
| PPO 在 LLM 中的瓶颈 | 需要同时加载 Actor + Critic + Ref + RM ≈ 4 个大模型 | 一辆车要同时塞 4 头大象——装不下 |
| 去掉 Critic | GRPO 不需要 Critic 网络，用组内均值做 Baseline | 不需要教练在场边打分了，球员们互相比就行 |
| 稀疏奖励 | 整条回答只有一个奖励分数（不逐 token 打分） | 厨师做完一整道菜才被评分，不是每切一刀都有分 |

## 二、关键公式卡片

### 组内归一化优势
$$\hat{A}_i = \frac{r_i - \text{mean}(r_1,\ldots,r_G)}{\text{std}(r_1,\ldots,r_G) + \epsilon}$$
> **人话**：同一道题生成 G 条回答，每条回答的优势 = (自己的分 - 组平均分) / 组内标准差。正数 = 比平均好，负数 = 比平均差。

### GRPO 完整 Loss
$$\mathcal{L}_{\text{GRPO}} = -\frac{1}{G} \sum_{i=1}^{G} \frac{1}{|y_i|} \sum_{t=1}^{|y_i|} \left[\min\left(r_{t,i} \hat{A}_i,\; \text{clip}(r_{t,i}, 1-\epsilon, 1+\epsilon) \hat{A}_i\right) - \beta \cdot \text{KL}_t\right]$$
> **人话**：这就是 PPO Clip Loss + 组内优势 + 每个 token 的 KL 惩罚 + 对序列长度归一化。

### 显存对比
$$\text{PPO} \approx 4 \times \text{模型大小}, \quad \text{GRPO} \approx 2\text{-}3 \times \text{模型大小}$$
> **人话**：GRPO 省掉一个 Critic，7B 模型能省 ~7B 参数的显存。

## 三、生动类比：高考排名制

想象高考数学阅卷：

- **PPO 的做法**（绝对评分 + 专家陪审）：
  - 每道题都请一位**专家（Critic）**先预估"这题应该得多少分"
  - 然后看学生实际得分和专家预估的差距
  - 问题：专家也是数学老师，光请专家就要多花一倍预算（**显存翻倍**）

- **GRPO 的做法**（组内排名 + 省掉专家）：
  - 不请专家了！
  - 同一道题让一组学生（G 个人）各自作答
  - 看每个学生"比组内平均分高多少"——**用组内比较代替专家预估**
  - 分数高于平均的 → 强化其答题策略
  - 分数低于平均的 → 压制其答题策略

> **核心洞察**：在 LLM 训练中，我们只需要知道"哪个好哪个差"（相对排名），不需要精确估计"到底值多少分"（绝对分数）。

## 四、GRPO 与数学推理的天然适配

假设一道数学题生成 8 条解答（G=8）：

| 第 i 条回答 | 答案 | 奖励 $r_i$ | 归一化优势 $\hat{A}_i$ |
|------------|------|-----------|---------------------|
| 回答 1 | 360 ✅ | 1.0 | +1.0 |
| 回答 2 | 350 ❌ | 0.0 | -1.0 |
| 回答 3 | 360 ✅ | 1.0 | +1.0 |
| 回答 4 | 39 ❌ | 0.0 | -1.0 |
| ... | ... | ... | ... |

- 正确答案（$r=1$）→ 组内优势 > 0 → 增强这种解题方式
- 错误答案（$r=0$）→ 组内优势 < 0 → 压制这种解题方式
- **完美映射**：不需要知道"360 这个答案值多少分"，只需要"对的比错的好"

## 六、PPO vs GRPO 终极对比

| 维度 | PPO | GRPO |
|------|-----|------|
| Critic | ✅ 需要（参数量 = Actor） | ❌ 不需要 |
| 优势估计 | GAE（依赖 $V(s)$） | 组内归一化（只看 $r_i$） |
| 显存 | ~4x 模型大小 | ~2x 模型大小 |
| 采样 | 1 条回答/prompt | G 条回答/prompt |
| 适合稀疏奖励 | 一般 | 非常适合 |
| 实现复杂度 | 高（需要维护 Critic） | 低（核心 ~20 行代码） |
| DeepSeek R1 | ❌ | ✅ |

## 七、一个特殊情况的思考

> **如果组内所有 G 条回答奖励都相同（全对或全错），会怎样？**

$\text{std} = 0$，$\hat{A}_i = 0$（所有优势为零），**这批数据不产生梯度**，相当于这道题白做了。

教训：G 要足够大，让组内有差异。实践中 G=16 是常见选择。

---

## 八、易混淆点 & 常见误区

1. **GRPO 仍然用 Clip！** 去掉的是 Critic，不是 Clip。Clip 仍然用于控制更新幅度。
2. **$\hat{A}_i$ 是 per-response 的，不是 per-token 的**：同一条回答的所有 token 共享同一个优势值。
3. **`log_probs_old.detach()` 是必须的**：旧策略是固定参考值，不能参与梯度计算。
4. **组内归一化 ≠ Batch Normalization**：BN 是对激活值跨 batch 归一化，GRPO 是对奖励在同一 prompt 的 G 条回答中归一化。

---

## 九、记忆口诀

> **"一题多答比高低，均值做线分优劣；不用教练省一半，Clip 和 KL 保安全"**
> - 一题多答 = 每个 prompt 生成 G 条回答
> - 均值做线 = 组内均值作 Baseline
> - 分优劣 = 归一化优势
> - 不用教练 = 去掉 Critic
> - 省一半 = 显存省约 25%~50%
> - Clip + KL = 沿用 PPO 的安全机制
