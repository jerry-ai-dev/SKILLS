## 一、核心概念速查表

| 概念 | 一句话定义 | 生活类比 |
|------|-----------|---------|
| 梯度累积 | 拆大 batch 为 K 小步，累积梯度后统一更新 | 搬砖：搬不动 100 块，就分 10 趟搬，效果一样 |
| Effective Batch Size | $N_\text{devices} \times B_\text{device} \times K_\text{accum}$ | 实际等效的大 batch 大小 |
| 混合精度训练 | 前向/反向用低精度(bf16)，参数更新用高精度(fp32) | 草稿纸上潦草算（快），誊写到作业本上工整（准） |
| fp32 | 4 字节，精度高，范围大 | A4 纸正常写——什么都写得下 |
| fp16 | 2 字节，精度中，范围小（易溢出） | 便签纸——省纸但写大数会写不下 |
| bf16 | 2 字节，精度较低，范围同 fp32 | 大号便签纸——省纸且写大数不会溢出 |
| Loss Scaling | fp16 专用，放大 loss 防止梯度下溢 | 蚂蚁太小看不见，用放大镜放大再操作 |
| GradScaler | PyTorch 自动 Loss Scaling 工具 | 那个自动调焦的放大镜 |

## 二、关键公式卡片

### 等效 Batch Size
$$\text{Effective Batch} = N_\text{devices} \times B_\text{device} \times K_\text{accum}$$
> **人话**：(几张卡) × (每张卡的 batch) × (累积几步) = 真正的 batch 大小。

### 梯度累积的关键一行
```python
loss = loss / accumulation_steps  # ← 不除的话梯度是 K 倍大！
loss.backward()
```
> **人话**：每小步的 loss 要除以 K，否则累积后的梯度是正常值的 K 倍。

### 显存粗估（bf16 混合精度 + Adam）
$$\text{显存} \approx \text{参数量} \times 14 \text{ bytes}$$
| 组件 | 大小 |
|------|------|
| 模型参数 (bf16) | params × 2B |
| 梯度 (fp32) | params × 4B |
| Adam 状态 m, v (fp32) | params × 8B |
| **合计** | **params × 14B** |

> **人话**：7B 模型 ≈ 7×10⁹ × 14 = **98 GB**（不含激活值），所以需要多卡。

## 三、生动类比

### 梯度累积 = 分批搬砖

工地要搬 100 块砖到对面（**大 batch**），但你一次只能搬 10 块（**小 batch**）：

- **笨方法**：找起重机搬 100 块（需要巨大显存）
- **聪明方法**：分 10 趟搬，每趟 10 块（梯度累积）
  - 每趟搬完**不砌墙**（不 `optimizer.step()`），先在工地堆着（梯度在 `.grad` 里累加）
  - 10 趟全搬完后再统一砌墙（`optimizer.step()`）
  - **关键**：每趟搬的砖要标记好"这只是 1/10"（`loss /= K`），否则砌墙时以为有 1000 块砖

### 混合精度 = 草稿纸 + 作业本

$$\underbrace{\text{前向 + 反向}}_{\text{草稿纸（bf16）}} \longrightarrow \underbrace{\text{参数更新}}_{\text{作业本（fp32）}}$$

- **草稿纸（bf16/fp16）**：算得快、用纸少（GPU 快、省显存）
- **作业本（fp32）**：最终答案要工整准确（参数精度不丢失）
- bf16 > fp16 的原因：bf16 的"纸张"更大（数值范围同 fp32），不会因为数字太大写不下（overflow）

## 四、fp16 vs bf16 决策树

```
要训练大模型？
├── GPU 支持 bf16（A100/H100/4090）？
│   ├── 是 → 用 bf16 ✅（不需要 GradScaler）
│   └── 否 → 用 fp16 + GradScaler
└── 只做推理？
    └── fp16 足够（不涉及梯度，不怕 underflow）
```

## 五、梯度累积代码模板（背下来）

```python
optimizer.zero_grad()
for step in range(K):
    x, y = get_mini_batch(step)
    loss = model(x, y)
    loss = loss / K           # ← 关键：归一化！
    loss.backward()           # 梯度自动累加在 .grad 里
# K 步累积完毕，统一更新
optimizer.step()
```

## 六、易混淆点 & 常见误区

1. **`loss /= K` 不能忘**：否则等效学习率被放大了 K 倍，训练爆炸。
2. **bf16 推荐不是因为精度高**：恰恰相反，bf16 精度比 fp16 低！推荐它是因为**数值范围大、不会溢出**。
3. **GradScaler 只在 fp16 时用**：bf16 天然不溢出，加 GradScaler 反而画蛇添足。
4. **14×参数量只是粗估**：没算激活值。实际显存还要看 batch size 和 seq_len。
5. **梯度累积不完全等价大 batch**：如果用了 BatchNorm（LLM 通常不用），BN 统计量在小 batch 上不准确。但 LLM 用 LayerNorm，没有这个问题。

## 七、实战估算练习

| 模型 | 参数量 | 最低显存（不含激活） | 适合什么卡 |
|------|--------|-------------------|-----------| 
| GPT-2 | 117M | ~1.6 GB | 任何 GPU |
| LLaMA-2 7B | 7B | ~98 GB | 2×A100-80G |
| LLaMA-2 70B | 70B | ~980 GB | 至少 16×A100 |

> **省显存三板斧**：① 梯度累积（减小 per-device batch）② bf16（参数减半）③ 梯度检查点（用计算换显存）

## 八、记忆口诀

> **"大 batch 装不下，分批来搬砖；除以 K 别忘了，否则梯度翻上天；fp16 怕溢出，bf16 更稳健"**
> - 分批搬砖 = 梯度累积
> - 除以 K = loss 归一化
> - bf16 稳 = 数值范围大
