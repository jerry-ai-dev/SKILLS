---

## 一、核心概念速查表

| 概念 | 一句话定义 | 生活类比 |
|------|-----------|---------|
| Tokenizer | 将文本拆分为 token（模型能处理的最小单元）的工具 | 把一篇文章拆成积木块，每块有个编号 |
| BPE (Byte Pair Encoding) | 从字符开始，反复合并最高频的相邻 token 对 | 打字缩写：经常打"lol"，就把 l+o+l 合并成一个快捷键 |
| 词级分词 (Word-level) | 每个单词是一个 token | 整本字典当词表——太大，而且遇到新词就傻眼 |
| 字符级分词 (Char-level) | 每个字母是一个 token | 按字母拆——词表小但序列巨长 |
| Byte-level BPE | 以字节(0-255)而非字符为基础单元 | 万能积木：任何语言的文字都能拆成 256 种基础积木的组合 |
| Special Token | 有特殊用途的控制 token（pad/eos/bos 等） | 标点符号：不是内容但帮你理解结构 |
| OOV (Out of Vocabulary) | 词表中没有的未知词 | 字典里查不到的网络新词 |

## 二、BPE 算法流程速记

### 训练阶段
```
1. 把所有单词拆成字符：hello → h e l l o </w>
2. 统计所有相邻 token 对的频率
3. 合并频率最高的那对：(l, l) → ll
4. 记录合并规则到列表
5. 重复 2-4 直到词表达到目标大小
```

### 推理阶段（对新词分词）
```
1. 把新词拆成字符
2. 按训练时的合并顺序优先级，逐步合并
3. 直到没有可合并的对
```

> **核心思想**：高频子词保持完整（效率高），低频词拆成子词组合（覆盖广），在字符和单词之间找平衡点。

## 三、生动类比：乐高积木工厂

想象你开了一家**乐高积木工厂**：

**字符级方案**（只卖基础件）：
- 只卖 26 种字母积木 + 标点积木
- 优点：库存只有 30 种零件
- 缺点：搭个"hello"要 5 块积木，大工程积木数量爆炸

**单词级方案**（卖成品模型）：
- 每个英语单词做一个成品积木
- 优点：搭东西快
- 缺点：库存 10 万种成品，新词（"vibe coding"）没有现成积木

**BPE 方案**（卖常用预制件 + 基础件）：
- 观察客户最常搭的组合（统计频率）
- 把高频组合做成预制件：un- / -tion / -ing / -ment
- 低频词用基础件拼
- **完美平衡**：词表适中（3w~10w），什么都能表示，常见词无需拆分

**Byte-level BPE**（万能基础件）：
- 基础件从"26 个字母"换成"256 个字节"
- 任何语言（中文、日文、emoji）都能处理——因为任何文字最终都是字节
- GPT-2、LLaMA、Qwen 都用这个

## 四、Special Tokens 速查

| Token | 作用 | 类比 |
|-------|------|------|
| `<pad>` | 补齐不等长序列 | 试卷答题纸的空白行——不算分 |
| `<eos>` / `</s>` | 标记序列结束 | 句号：看到这里就停 |
| `<bos>` / `<s>` | 标记序列开始 | 开头引号 |
| `<\|user\|>` | 标记用户输入开始 | 对话中的"客户说：" |
| `<\|assistant\|>` | 标记模型回答开始 | 对话中的"客服答：" |
| `<\|think\|>` | DeepSeek R1 的思维链标记 | 脑中的"让我想想……" |

> **关键**：`<pad>` 对应的 label 设为 `-100`——这和 Lesson 7 的 Loss Masking 衔接！

## 五、BPE 合并示例（手动推导）

语料："hello hello hello world"

```
初始: h e l l o </w>    (频率3)
      w o r l d </w>    (频率1)

Step 1: 最高频对 (l, l) → ll
      h e ll o </w>     (频率3)
      w o r l d </w>    (频率1)

Step 2: (h, e) → he    (频率3)
      he ll o </w>      (频率3)

Step 3: (he, ll) → hell (频率3)
      hell o </w>       (频率3)

Step 4: (hell, o) → hello (频率3)
      hello </w>        (频率3)
```

> 经过 4 次合并，"hello" 变成了一个完整 token！因为它出现频率高。而 "world" 出现少，合并轮次多才会被合成一个 token。

## 六、三种分词方案的终极对比

| 方案 | 词表大小 | 序列长度 | OOV 问题 | 适合场景 |
|------|---------|---------|---------|---------|
| 字符级 | ~30 | 非常长 | 无 | 简单实验 |
| 词级 | ~100k+ | 短 | 严重 | 传统 NLP |
| **BPE** | **30k~100k** | **适中** | **无** | **现代 LLM** |
| **Byte-level BPE** | **同上** | **适中** | **无** | **多语言 LLM** |

## 七、易混淆点 & 常见误区

1. **BPE 是统计方法不是语言学方法**：它完全基于频率，不关心词根、词缀等语言学规则。
2. **合并是有顺序的**：推理时必须按训练时记录的合并顺序（优先级）来合并，不能随便合。
3. **`Ġ` 前缀**：GPT-2 的 tokenizer 用 `Ġ` 表示空格前缀，如 `Ġworld` = ` world`（前面有空格）。
4. **中文在 Byte-level BPE 中**：一个汉字通常被编码为 3 个字节 token（UTF-8 编码），所以中文序列在英文 tokenizer 中会很长。
5. **词表大小是超参数**：合并次数决定词表大小。合并越多 → 词表越大 → 序列越短 → 但模型嵌入层越大。

## 八、记忆口诀

> **"高频合并成一块，低频拆分拼起来；字节做底通百语，Special Token 管格式"**
> - 高频合并 = BPE 合并高频对
> - 低频拆分 = 稀有词用子词拼
> - 字节做底 = Byte-level BPE
> - Special Token = pad/eos/bos 等
